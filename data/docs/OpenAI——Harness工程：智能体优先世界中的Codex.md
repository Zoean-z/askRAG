# Harness工程：智能体优先世界中的Codex

> 来源：[OpenAI Engineering Blog](https://openai.com/index/harness-engineering/) — Ryan Lopopolo, 2026-02-11

---

## 背景

团队用5个月时间，从空仓库开始，**0行手写代码**，用Codex（GPT-5驱动）构建了一个有内部日活用户的产品。约100万行代码、~1500个PR、3→7人工程团队、人均3.5 PR/天。

核心约束：**人类操舵，智能体执行（Humans steer, Agents execute）**。

---

## 核心框架

### 1. 工程师角色重定义

工程师不再写代码，而是：
- 设计环境（environment design）
- 指定意图（intent specification）
- 构建反馈循环（feedback loops）

当智能体失败时，正确的反应不是"再试一次"，而是问：**"缺什么能力？如何让这个能力对智能体可读且可执行？"**

### 2. 渐进式披露的上下文管理

**失败方案**：把所有指令塞进一个巨大的 AGENTS.md
- 上下文是稀缺资源，巨型文件挤掉了任务本身
- "一切都重要"等于"什么都不重要"
- 单体文件会腐烂且无法机械验证

**成功方案**：AGENTS.md（~100行）只做**目录**，指向结构化的 `docs/` 体系
```
AGENTS.md          ← 地图（~100行）
ARCHITECTURE.md    ← 领域和分层
docs/
├── design-docs/   ← 设计文档（有索引、核心信念）
├── exec-plans/    ← 执行计划（活跃/完成/技术债）
├── product-specs/ ← 产品规格
├── references/    ← 外部参考（llms.txt格式）
└── *.md           ← 设计/前端/安全/可靠性等
```

配合 linter + CI 验证知识库的时效性和交叉引用完整性。"文档园丁"智能体定期扫描过时文档并提PR修复。

### 3. 应用可读性（Application Legibility）

让智能体能直接感知和操作应用本身：
- 每个 git worktree 可独立启动应用实例
- Chrome DevTools Protocol 集成：DOM快照、截图、导航
- 本地可观测性栈：日志(LogQL)、指标(PromQL)、追踪(TraceQL)
- 智能体可以自驱动验证："确保服务启动在800ms内"

**关键洞察**：智能体看不到的东西等于不存在。Slack讨论、Google Docs、人脑中的知识——如果不在仓库里，对智能体就是不可见的。

### 4. 架构执行：约束而非微管理

**强制不变量，不指定实现方式**。例如：
- 要求在边界处解析数据形状（parse, don't validate），但不指定用哪个库
- 每个业务领域固定分层：Types → Config → Repo → Service → Runtime → UI
- 跨切关注点（auth/telemetry/feature flags）通过唯一接口 Providers 注入
- 自定义 linter 强制执行，且 **错误信息本身包含修复指令**（注入到智能体上下文中）

> 在人类优先的工作流中，这些规则可能显得迂腐。对智能体来说，它们是乘数。

### 5. 熵与垃圾回收

智能体会复制仓库中已有的模式——包括不好的模式。漂移不可避免。

**失败方案**：每周五人工清理"AI slop"（占20%工时，不可扩展）
**成功方案**：编码"黄金原则" + 定期后台Codex任务扫描偏差 → 自动提交重构PR → 大多数1分钟内可审查并自动合并

> 技术债就像高息贷款：持续小额偿还 < 积累后痛苦清算。

### 6. 自主性阶梯

最终状态——单个prompt可驱动完整特性周期：
验证代码库状态 → 复现bug → 录制失败视频 → 实现修复 → 验证修复 → 录制成功视频 → 开PR → 回应反馈 → 修复构建失败 → 仅在需要判断时升级到人类 → 合并

## 与库内文档的连接

- **渐进式披露** 直接验证了 `HARNESS_ENGINEERING_GUIDE` 的"file-based state"原则——OpenAI用同样方法但规模更大
- **AGENTS.md作目录** 是本库 INDEX 应采用的模式
- **"缺什么能力就补什么能力"** 与 `Sprint1_根因地图` 的"干预切入点"思路同构
- **黄金原则+自动垃圾回收** 是 `HARNESS_ENGINEERING_GUIDE` 中"每5个feature做质量审计"的升级版
- **约束而非微管理** 印证了 `杠杆·结构·奇点` 的核心论点：结构 > 执行
- **应用可读性** 拓展了 Harness 的边界——不只是代码生成，还包括运行时感知
